前端单页面应用开发规范化要点

posted 2020-01-15 in Vue ， 规范 count： 7.8k readingTime ≈ 7 分钟

前言

　　经过几个月时间的开发和试运营，公司项目逐渐走向规范化并趋于成熟，终于不用重复的搬运代码，趁着现在有时间，整理一下前端单页面应用开发规范化的要点。
　　适当的规范和标准不是为了消灭小姐姐小哥哥们代码内容的创造性、优雅性，而是限制过度个性化，以一种普遍认可的统一方式一起做事，提升协作效率。所谓前人栽树后人乘凉，提高我们代码的可读性和可维护性，是我们共同努力的目标。
　　具体知识点在项目中的常用操作参考以下链接：

  • Element
  • Vant
  • CSS预处理器
  • ESLint常用配置
  • 钉钉免登录
  • 高德地图
  • ECharts

项目技术栈

  • 电脑端：Vue + Element + Sass/Less；
  • 移动端（钉钉/微信）：Vue + Vant/Mint UI + Sass/Less；
  • 微信小程序：uni-app + uni ui /uni衍生的插件市场 + Sass/Less；
  • 项目涉及的可视化模块：地图部分使用高德地图，图表部分使用ECharts可视化库。

文件规范化

文件存放位置规范

  1. icons：图标库，iconfont引入阿里图标库；svg是项目的svg图标存放位置；
  2. styles：样式文件，一般使用Sass语言，若要使用其他语言可在该文件下建文件夹，比如新建less文件夹，内部放置less文件；
  3. api：接口文件，放置所有的接口，request.js文件放在api文件夹根目录，用于处理axios初始配置；按照模块名新建文件夹，存放该模块下的所有子模块接口文件；接口文件命名规则遵循跟views的文件名一致，方便后人查看翻阅；调用规则遵循 @api/模块名/子模块名；公共接口存在放api/commonApi/index.js里面，登录和退出接口也放在这里；
  4. utils：方法文件，方法按照处理类型分类，比如validate表示存放所有验证的方法，time处理所有涉及时间的方法，以此类推；所有的方法文件都在utils/index.js中引入并导出；调用时通过@utils调用即可；
  5. constant：变量文件，按照常规和非常规常量进行分类；常规的都放在commonConstant；非常规可新建文件，比如地图相关的所有放mapConstant,图表相关放echartsConstant；所有的变量文件都在constant/index.js中引入并导出；调用时通过@constant调用即可；
  6. router：路由文件，所有功能的路由文件都在router内；在router/modules/下新建模块的路由文件；文件命名规则遵循跟views的文件名一致；所有的路由文件都在router/index.js中引入并导出为路由对象；其中constantRoutes为固定路由，asyncRoutes为动态路由。

文件命名规范

　　在vue官方规范基础上结合大家的编码习惯独家定制：

  1. Component：所有的Component文件都是以大写开头 (PascalCase)，包括views下的Component文件，但除了index.vue；
  2. JS：所有的.js文件遵循驼峰命名(kebabCase)；
  3. Views：views下的文件夹和单组件文件都遵循驼峰命名(kebabCase)。

代码规范

  1. 文本缩进：一次缩进两个空格
  2. 对象命名：
    • 常量：采用全大写并且单词以_分割(KEBAB_CASE)
    • 函数名：
      • 类（构造函数）：以大写开头的驼峰命名格式 (PascalCase);
      • 函数：首字母小写，驼峰命名（kebabCase）。
    • 变量名：首字母小写，驼峰命名(kebabCase)，匈牙利命名。
      匈牙利命名法语法：变量名＝变量类型＋对象描述，例如：
      起名类型 命名前缀 举例
      Array数组 a aList,aGroup
      Boolean逻辑 b bChecked,bHasLogin
      Function 函数 f fGetHtml,fInit
      Integer 数字 n nPage,nTotal
      Object 对象 o oButton,oDate
      Regular E正则 r rDomain,rEmail
      String 字符 s sName,sHtml
  3. 代码注释
    原则上：如无必要，勿增注释，如有必要，尽量详尽。以此提高代码可读性和可维护性。注释分为单行注释：//和多行注释：/**/。 参考文档：JSDoc Guide
  4. 代码检查
    • 建议VSCode配置ESLint插件：每次保存，VSCode就能标红不符合 eslint 规则的地方，同时还会做一些简单的自我修正。另外运行npm run lint -- --fix，eslint会自动修复一些简单错误。
    • 一般所有的配置项都存放在 .eslintrc.js 中，如果觉得配置过于严格可自行修改内部配置；原则上为了减少代码不必要的错误使用最严格的plugin:vue/recommended来校验代码。
    • 配置步骤：
      • VSCode扩展插件中搜索安装eslint插件；
      • 在根目录配置.eslintrc.js文件；
